## Objetivo

Implementar a estrutura de multitenancy adicionando a tabela `companies` e relacionando todos os dados existentes ao contexto de empresa. Garantir isolamento total de dados entre empresas via RLS.

## Contexto

O sistema atual nao tem isolamento por empresa — todos os dados sao compartilhados entre usuarios autenticados. Para um SaaS, cada empresa (vidracaria/fabrica) deve ter seus proprios dados completamente isolados.

- [x] AC3: Tabelas principais tem coluna `company_id`: clientes, fornecedores, produtos, orcamentos, pedidos, servicos, planos_de_corte
- [x] AC4: RLS em todas as tabelas usa `company_id` para isolamento
- [x] AC5: Funcao `get_user_company_id()` retorna o company_id do usuario autenticado

- [x] AC7: Admin pode ver dados apenas da sua empresa
- [x] AC8: Zero dados vazando entre empresas (teste de isolamento)
- [x] AC9: Migration backward-compatible (dados existentes sao mantidos)

## Schema

```sql
-- Nova tabela
CREATE TABLE companies (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  nome TEXT NOT NULL,
  cnpj TEXT UNIQUE,
  email TEXT,
  telefone TEXT,
  plano TEXT NOT NULL DEFAULT 'basico'
    CHECK (plano IN ('basico', 'essencial', 'avancado')),
  status TEXT NOT NULL DEFAULT 'ativo'
    CHECK (status IN ('ativo', 'inativo', 'suspenso')),
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- RLS Example
CREATE POLICY "company_isolation" ON clientes
  USING (company_id = get_user_company_id());
```

### Achados

- **CONCERN (SEGURANÇA):** RLS atualizada apenas para `clientes`, `orcamentos`, `pedidos`. As tabelas `fornecedores`, `produtos`, `servicos`, `planos_de_corte` têm `company_id` adicionado mas **sem novas policies** de isolamento. Risco de vazamento de dados entre empresas nessas tabelas se as policies anteriores não cobrirem `company_id`. Recomendado: verificar/criar policies para essas tabelas em migration complementar.
- **INFO:** `agenda_eventos` recebeu `company_id` mas sem RLS correspondente na migration.
- **INFO:** `companies` não tem RLS para INSERT por usuário comum — apenas SELECT/ALL via policies criadas. Correto pois signup cria company via trigger (SECURITY DEFINER), não via usuário diretamente.

### Conclusão

Fundação multitenancy sólida. RLS incompleta para 4 tabelas é concern de segurança que deve ser adressado em story de follow-up antes de go-live.
