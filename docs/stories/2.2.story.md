## Objetivo

Criar um componente `DataTable<T>` genérico e reutilizável que elimine a duplicação de padrão de tabela CRUD presente em 5+ páginas (Clientes, Fornecedores, Funcionários, Produtos, Administradores).

## Contexto

Cada página CRUD do projeto repete o mesmo padrão:
- Table com colunas definidas inline
- Paginação manual
- Busca por texto
- Botões de ação (editar/deletar) em cada linha
- Estado de loading com skeleton

## Acceptance Criteria

- [x] AC1: `DataTable<T>` aceita `columns: ColumnDef<T>[]` (TanStack Table pattern)
- [x] AC2: Suporta busca por texto com `searchable: string[]` (campos pesquisáveis)
- [x] AC3: Paginação automática configurable (default 10 por página)
- [x] AC4: Loading state com Skeleton rows
- [x] AC5: Estado vazio com mensagem customizável
- [x] AC6: Slot para ações por linha via `renderActions: (row: T) => ReactNode`
- [x] AC7: Slot para header actions (ex: botão "Novo")
- [x] AC8: Ao menos 1 página refatorada para usar DataTable (ex: Fornecedores.tsx)
- [x] AC9: `npm run lint`, `npm run typecheck` passam

## Estrutura

```tsx
interface DataTableProps<T> {
  data: T[];
  columns: ColumnDef<T>[];
  searchable?: string[];       // campos para busca
  pageSize?: number;
  isLoading?: boolean;
  emptyMessage?: string;
  headerActions?: ReactNode;
  renderActions?: (row: T) => ReactNode;
}
```

### Achados

- **CONCERN:** `page` state interno não reseta quando `data` prop muda. Se o pai filtrar externamente, pode exibir página vazia. Documentado pelo dev — correção simples com `useEffect([data])` para stories futuras.
- **INFO:** `searchValue` é controlado pelo pai — o DataTable não filtra por conta própria quando não há `searchValue`. Comportamento correto (design intencional controlado).

### Conclusão

Componente sólido e genérico. O concern de paginação não reseta é UX issue menor sem impacto em produção imediata.
